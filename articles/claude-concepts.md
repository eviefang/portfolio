最近在系统研究 Claude 的产品体系，从 MCP 到 Skill，到 Plugin，到 Connector，再到它们之间的协作机制。一路看下来，最有收获的不是"学到了新概念"，而是发现——我之前在阿里实习时做过的一些架构决策，在今天的行业标准里原来有了对应的名字。

这篇是一次自我校准的学习笔记。不是概念科普（网上已经够多），而是一个求职者用新的概念框架重新看自己做过的事：哪些是做对了的、哪些现在可以升级、哪些当年想都没想到。

## 一、先把概念坐标系对齐

在展开之前，先花 30 秒把四个核心概念的定位摆清楚：

| 层级 | 解决什么问题 | 类比 |
|---|---|---|
| MCP | 通信协议 | USB-C 接口标准 |
| Connector | 单点能力接入 | 外接设备 |
| Skill | 场景方法论 | 操作手册 |
| Plugin | 打包分发 | 整套工作套装 |

一句话总结：**MCP 是管道，Connector 是接入点，Skill 是方法论，Plugin 是把前三样打包成给特定角色用的套件。**

这四层不是竞品关系，是分层协作关系。理解这一点，再往下看才有意义。

## 二、单 Agent 架构下，模型是不能切换的

很多人以为 Agent 能"根据请求复杂度自动切换模型"——简单问题用小模型快速响应，复杂问题用大模型深度思考。我最初也这么以为。

但仔细想会发现：**一次 API 调用绑定一个模型，这是物理约束，不是产品选择。** 就像你在 Claude 客户端里选定了模型后，整段对话都由它完成，中途无法更换。

所以"按复杂度切模型"这件事，必然跨出单 Agent 架构，进入多 Agent 编排——需要一个前置的 Router Agent 做分流，然后把任务派发给不同模型的执行 Agent。

这个边界对做 Agent 产品的人很重要：

| 能力 | 单 Agent 能做 | 多 Agent 才能做 |
|---|---|---|
| 动态加载不同 Skill | ✅ | - |
| 动态调整 Tool 调用次数 | ✅ | - |
| 动态调整上下文长度 | ✅ | - |
| **动态切换模型** | ❌ | ✅ |

前三项本质上是"**喂给同一个模型的内容变化**"——模型不变，输入变。而模型切换是"**执行主体变化**"——必须让另一个 Agent 接手。两者是完全不同的架构动作。

这个区分看起来细，但它决定了你在设计一个 Agent 系统时，复杂度预算应该花在哪里。大多数业务场景根本不需要跨模型调度，单 Agent 内的 Skill/Tool/上下文分级就够用了——盲目上多 Agent，只会把工程复杂度放大好几倍。

## 三、后端搭舞台，Claude 演戏

搞清楚了架构边界，下一个问题是：**在单 Agent 系统里，"谁决定什么"？**

这是我觉得整个学习过程中最有方法论价值的一节。能把这张表背下来，很多 AI 产品设计问题会变得清晰很多：

| 决策 | 谁来决定 | 发生时机 |
|---|---|---|
| 用哪个模型 | 后端代码 | API 调用前 |
| 加载哪些 Skill 作为候选 | 后端代码 | API 调用前 |
| 实际使用哪个 Skill | Claude | 对话中 |
| 调用哪个 Tool | Claude | 对话中 |
| Tool 参数怎么填 | Claude | 对话中 |

可以提炼出一条原则：

> **越底层、越物理、越需要权限控制的决策，越归后端；越上层、越语义、越需要理解的决策，越归模型。**

这条原则解决的是 AI 产品设计中一个被严重低估的问题——很多团队在设计 AI 系统时走两种极端：要么把什么都交给模型（系统不可控、安全边界模糊），要么把什么都写死在代码里（系统不灵活、用户体验僵硬）。

一个好的 Agent 系统，应该让后端负责"物理层"和"权限层"的决策（能用什么模型、能访问哪些 Skill、能调用哪些 Tool），让模型负责"语义层"的决策（在给定的舞台上怎么理解、怎么选择、怎么生成）。

这个比喻可以再提炼一下：**后端搭舞台，Claude 演戏。**

## 四、用这个框架，重看我在阿里做过的事

我在阿里实习时做过一个对话 Agent，当时为了控制 Prompt 膨胀和方便 Badcase 归因，在主 System Prompt 里设计了一套"简单/复杂/工具型"的请求分流机制。

（Badcase 归因：把模型输出的错误，定位到是 Prompt 层写得不好、工具层执行出错、还是路由层判断偏差——每一类错误归因对应不同的修复动作。）

用今天的概念框架回头看，当时的一些决策至今依然成立：

- **选 MCP Server 架构**——把业务能力和对话逻辑解耦，业务接口独立演进，不被 Prompt 绑架
- **Tool 粒度拆分**——每个工具职责单一，Badcase 归因能精确定位到工具层
- **用 JSON Schema 约束 Tool 调用**——可验证、可观测，减少模型"自由发挥"带来的线上事故

但也有明显可以升级的地方。最核心的一点：**当年我把请求分流逻辑硬编码在主 Prompt 里，今天更合适的做法是拆成独立的 Skill 文件**——每个业务场景一个 Skill，Claude 根据 Skill 的 description 自主判断加载哪一个。这样做的收益是三层的：

第一，System Prompt 可以大幅瘦身，只保留角色、语气、全局约束和顶层路由，信号更集中；

第二，每个 Skill 可以独立迭代、独立评测、独立灰度，Badcase 归因从"归因到 Prompt 段落"升级到"归因到 Skill 模块"；

第三，Skill 文件可以交给产品/运营直接维护，不必每次改业务流程都走研发排期。

但这里有个关键诚实要讲——

**我当年做这个分流机制时，Anthropic 还没推出 Skill 这个概念。** 我面对的问题很具体：Prompt 越写越长、业务场景越加越多、Badcase 越来越难归因。解决思路是把不同类型的请求路由到不同处理分支，每个分支用更精简的指令集。

现在回头看，这其实就是 Skill 化的雏形。不是说我当年就有架构远见，而是真实工程约束下，只要做得足够深、足够认真，往往会独立摸索到后来被行业标准化的方向。

这个发现对我的意义是：**我确认了自己当时的思考方向是对的，也确认了我现在学的新概念不是空中楼阁**——而是给早期摸索者的工作一个更清晰的语言。

## 五、这次复盘对我找 AI PM 岗位的意义

AI PM 岗位的候选人里，能背出 MCP、Skill、Plugin 定义的人不少，但能分清这些概念背后架构边界的人不多。

能穿透概念表面看到本质区别——单 Agent 和多 Agent 的分界在哪、物理层决策和语义层决策的分工在哪、Skill 和 Tool 的职责差异在哪——是比"记住新名词"重要得多的能力。新概念不会取代实战经验，而是给有实战经验的人**一套重新看自己工作的语言**。

> 这次系统学习最大的收获不是学到了什么新东西，而是确认了一件事——**自己一直在做对的事情，现在有了正式的名字。**
